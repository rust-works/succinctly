# YAML 1.2 Compliance: The Document from Hell

[Home](../../../) > [Docs](../../) > [Compliance](../) > YAML 1.2

This document demonstrates how `succinctly yq` handles the infamous YAML pitfalls
described in [The YAML Document from Hell](https://ruudvanasseldonk.com/2023/01/11/the-yaml-document-from-hell).

It covers **scalar type resolution** only. For structural conformance measured against the

Plain-scalar resolution is implemented once, in `src/yaml/scalar.rs`

leading-dot forms `.inf`/`.nan` to be floats, so the bare words stay strings.

## Detailed Analysis

### 1. Sexagesimal Numbers (The Time Trap)

**YAML 1.1 Problem:** Values like `22:22` were interpreted as base-60 (sexagesimal) numbers.
`22:22` = 22Ă—60 + 22 = 1342.

**YAML 1.2 Solution:** Sexagesimal support was removed. Colon-separated values are strings.

```bash
$ echo 'time: 22:22' | succinctly yq -o json '.'
{
  "time": "22:22"
}
```

âś… **succinctly:** Correctly treats `22:22` as string `"22:22"`.

### 2. The Norway Problem

**YAML 1.1 Problem:** `no` (and `yes`, `on`, `off`, etc.) were interpreted as booleans.
Country code `NO` for Norway became boolean `false`.

**YAML 1.2 Solution:** Only `true/True/TRUE` and `false/False/FALSE` are booleans.

```bash
$ echo 'countries: [us, gb, no, fr]' | succinctly yq -o json '.'
{
  "countries": ["us", "gb", "no", "fr"]
}
```

âś… **succinctly:** Correctly treats `no` as string `"no"`.

### 3. Boolean Keys

**YAML 1.1 Problem:** Map keys like `on:` became boolean `true`.

**YAML 1.2 Solution:** Keys are not subject to type coercion.

```bash
$ echo 'on: enabled
off: disabled' | succinctly yq -o json '.'
{
  "on": "enabled",
  "off": "disabled"
}
```

âś… **succinctly:** Keys remain strings.

### 4. Version Numbers (The Float Trap)

**The Issue:** `3.10` looks like a version but parses as float `3.1` (trailing zero dropped).

**YAML 1.2 Spec:** This is **correct behavior**. The Core Schema float regex matches decimal numbers:
```
[-+]? ( \. [0-9]+ | [0-9]+ ( \. [0-9]* )? ) ( [eE] [-+]? [0-9]+ )?
```

**Solution:** Quote version numbers to preserve them as strings.

```bash
# Unquoted - becomes float (spec-compliant)
$ echo 'python: 3.10' | succinctly yq -o json '.python'
3.1

# Quoted - preserved as string
$ echo 'python: "3.10"' | succinctly yq -o json '.python'
"3.10"
```

âś… **succinctly:** Correctly implements YAML 1.2 spec. Quote versions to preserve them.

### 5. Quoted String Preservation

**The Issue:** Quoted values should remain strings regardless of content.

```bash
$ echo 'value: "true"' | succinctly yq -o json '.value'
"true"

$ echo 'value: "42"' | succinctly yq -o json '.value'
"42"

$ echo 'value: "3.10"' | succinctly yq -o json '.value'
"3.10"
```

âś… **succinctly:** Quoted values are always strings (P10 Type Preservation optimization).

### 6. Boolean Recognition

**YAML 1.2 Core Schema:** Only these values are booleans:
- `true`, `True`, `TRUE`
- `false`, `False`, `FALSE`

```bash
$ echo 'a: true
b: True
c: TRUE
d: false
e: False
f: FALSE' | succinctly yq -o json '.'
{
  "a": true,
  "b": true,
  "c": true,
  "d": false,
  "e": false,
  "f": false
}
```

âś… **succinctly:** Recognizes all six boolean spellings.

### 7. Null Recognition

**YAML 1.2 Core Schema:** These values are null:
- `null`, `Null`, `NULL`
- `~`
- Empty value

```bash
$ echo 'a: null
b: Null
c: NULL
d: ~
e:' | succinctly yq -o json '.'
{
  "a": null,
  "b": null,
  "c": null,
  "d": null,
  "e": null
}
```

âś… **succinctly:** Recognizes all five spellings ([#226](https://github.com/rust-works/succinctly/issues/226)).
Other casings (`NuLl`, `nULL`) are not in the core schema and stay strings, matching `yq`.

### 8. Tags (Unsupported)

**YAML Feature:** Tags like `!custom` or `!!str` provide explicit type hints.

In block context, a tag is a parse error:

```bash
$ echo 'value: !custom data' | succinctly yq '.'
Error: YAML parse error: tags (!) not supported at offset 7
```

In flow context it is silently absorbed into the scalar instead:

```bash
$ echo 'a: [!!str x]' | succinctly yq -o json -I 0 '.'
{"a":["!!str x"]}
```

âš ď¸Ź **succinctly:** Tags are not supported, and the two contexts disagree about how to
fail. See [Known Limitations](limitations.md#tags--33-cases-31-load-2-parse).

## Differences from System yq

All remaining scalar-resolution differences are deliberate (measured against pinned
yq v4.53.3 with `-o=json`):

| Input                  | succinctly     | mikefarah/yq   | Why succinctly differs                     |
|------------------------|----------------|----------------|--------------------------------------------|
| `.inf` / `.nan` â†’ JSON | `null`         | Error          | JSON has no inf/nan; erroring is unhelpful |
| `.inf \| tag`          | `!!null`       | `!!float`      | DOM path loses non-finite floats (#222)    |
| `1_000`                | `"1_000"`      | `1000`         | YAML 1.1 legacy; 1.2 core says string      |
| `0X2A` / `0O52`        | string         | `42` / Error   | 1.2 core requires lowercase `0x`/`0o`      |
| `0b101` (binary)       | `"0b101"`      | Error          | Not in 1.2 core; yq errors on JSON output  |
| `-0x2A` (signed hex)   | `"-0x2A"`      | Error          | 1.2 core forbids signs on based ints       |
| `0xFFFFFFFFFFFFFFFF`   | string         | Error          | Overflows `i64`; yq errors on JSON output  |
| `-0`                   | `0`            | `-0.0`         | go-yaml quirk (signed-zero float)          |
| `18.0` (compact `-I=0`)| `18`           | `18.0`         | Float formatting, tracked in #169          |

The `Error` rows are yq resolving a scalar as `!!int`/`!!float` and then failing to
serialize it (`strconv` rejects the source text) â€” byte-parity there is impossible, so
succinctly keeps these as strings per the 1.2 core schema instead.

**Notes:**
- Plain scalars resolve per the YAML 1.2 core schema (`src/yaml/scalar.rs`, #226):
  `null`/`Null`/`NULL`/`~`/empty, the six bool spellings, decimal/hex/octal ints,
  and floats including the leading-dot `.inf`/`.Inf`/`.INF`/`.nan` family (with
  optional sign on infinities). Bare `nan`/`inf`/`Infinity` are strings.
- succinctly's pretty JSON output preserves `.0` in floats for round-trip fidelity;
  the compact (`-I=0`) streaming path reformats parsed numbers.

## Recommendations

1. **Always quote version numbers:** `python: "3.10"` not `python: 3.10`
2. **Always quote country codes:** `country: "NO"` not `country: NO`
3. **Use explicit booleans:** `enabled: true` not `enabled: yes`
4. **Avoid tags:** succinctly does not support YAML tags
