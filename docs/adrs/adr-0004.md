# ADR-0004: Reject Software Prefetching for Large YAML Files (P2.6)

## Status

✅ Accepted

## Context

The YAML oracle (ADR-0003) is a sequential scan over the input, so its throughput is partly
bound by memory latency on large files. A natural idea (tracked as optimization **P2.6**) was to
issue explicit software prefetch hints (`_mm_prefetch` with `_MM_HINT_T0`, at a 256-byte
distance) ahead of the scan cursor, to hide memory latency and speed up parsing of large
documents.

## Decision

We will **not** add software prefetching to the YAML parser.

Benchmarking the change (AMD Ryzen 9 7950X, prefetch inserted at four loop sites) showed no
benefit on small inputs and a large regression on exactly the large-file case it was meant to
help:

| Size  | Change   |
|-------|----------|
| 1 KB  | +0.5%    |
| 10 KB | +3.7%    |
| 100 KB| +0.8%    |
| 1 MB  | **+30.3%** (2.58 ms → 3.36 ms) |

Modern CPU **hardware** prefetchers already track the sequential access pattern of the scan and
prefetch more effectively than manual hints. The manual prefetches instead caused cache
pollution and interfered with the hardware prefetcher, so the 1 MB case — the target of the
optimization — got 30% slower. Rejected 2026-01-17.

Full analysis: [parsing/yaml.md § P2.6](../parsing/yaml.md#p26-software-prefetching-for-large-files---rejected-).

## Consequences

- The parser relies on the hardware prefetcher for sequential access, keeping the hot loop free
  of prefetch intrinsics and their tuning parameters (hint level, distance).
- Establishes an empirical guideline for this codebase: **do not add software prefetching for
  sequential parsing** — hardware prefetchers win. This was the first of several optimizations
  where micro-level intuition was contradicted by end-to-end measurement (see ADR-0005, ADR-0006).
- Negative: none of substance — the rejected code was reverted. The cost was the investigation
  time, preserved here so the experiment is not repeated.
