# ADR-0005: Reject SIMD Threshold Tuning (P2.8)

## Status

✅ Accepted

## Context

Several of the YAML oracle's inner routines dispatch to SIMD only when the region to scan is
long enough to amortize the setup cost (e.g. `skip_spaces_simd`, `find_quote_or_escape`,
`find_single_quote`). Optimization **P2.8** proposed tuning these length thresholds — lowering
or introducing them (skip_spaces 8→12, adding 16-byte thresholds to the quote scanners) — because
micro-benchmarks of the routines in isolation suggested a 2–4% gain.

## Decision

We will **not** tune the SIMD dispatch thresholds.

End-to-end benchmarks contradicted the micro-benchmarks outright — the change **regressed**
quoted-string workloads:

| Workload            | Change  |
|---------------------|---------|
| quoted/double/10    | +10.5%  |
| quoted/single/10    | +15.5%  |
| quoted/double/100   | +8.0%   |
| quoted/single/100   | +8.1%   |
| quoted/double/1000  | +3.0%   |

On modern CPUs, branch prediction and inlining make the scalar fast path for short strings
essentially free, so raising the SIMD threshold added overhead without a corresponding win, and
the isolated micro-benchmark did not capture the surrounding context (inlining, branch history).
All three thresholds were reverted. Rejected 2026-01-17.

Full analysis: [parsing/yaml.md § P2.8](../parsing/yaml.md#p28-simd-threshold-tuning---rejected-).

## Consequences

- The existing SIMD thresholds are left as-is.
- Reinforces the guideline from ADR-0004 and generalizes it: **micro-benchmark wins do not imply
  end-to-end wins**, and threshold tuning in particular is counterproductive on modern CPUs.
  This is the second of three consecutive rejections (P2.6, P2.8, P3) driven by the same lesson,
  which later justified rejecting P5 at the analysis stage (ADR-0007).
- Negative: none — the change was reverted; the value is the recorded evidence.
