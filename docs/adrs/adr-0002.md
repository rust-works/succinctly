# ADR-0002: Table-Driven PFSM as the Portable JSON Parser

## Status

✅ Accepted

## Context

Building the JSON semi-index (ADR-0001) requires classifying every byte of the document to find
structural characters and string boundaries, while tracking whether the scanner is in-JSON,
in-string, in-escape, or in-value. This classification is the hot loop of JSON indexing — it runs
over every input byte — so its throughput sets the ceiling for JSON performance.

The `json/` module offers several strategies: a 3-state `simple` cursor, a 4-state `standard`
cursor, per-architecture SIMD classification (AVX2/SSE4.2 on x86_64, NEON on aarch64), a scalar
reference implementation, and a precomputed-lookup-table **PFSM** (Parallel Finite-State Machine).
We needed to decide (a) how the portable, no-SIMD path parses, and (b) how `JsonIndex::build()`
dispatches across platforms.

## Decision

We will build the JSON semi-index with the 4-state **standard cursor** state machine, and
implement its **portable (non-SIMD) path with a table-driven PFSM** rather than scalar branching.

`JsonIndex::build()` dispatches by target architecture
([src/json/light.rs](../../src/json/light.rs)):

- On **x86_64** (AVX2 › SSE4.2 › SSE2) and **aarch64** (NEON), it runs SIMD character
  classification feeding the standard cursor ([src/json/simd/](../../src/json/simd/)). This is the
  default on essentially all production targets.
- On **other platforms**, it calls [`standard::build_semi_index`](../../src/json/standard.rs),
  which — per its own comment — "always uses PFSM optimized … [because] it's the fastest and
  doesn't require special CPU features." The PFSM replaces per-byte branching with 256-entry
  lookup tables (`TRANSITION_TABLE` / `PHI_TABLE`) that pack all four state transitions into a
  single `u32`. Tables live in [src/json/pfsm_tables.rs](../../src/json/pfsm_tables.rs) and the
  processor in [src/json/pfsm_optimized.rs](../../src/json/pfsm_optimized.rs).

The PFSM is drawn from Mytkowicz et al., *Data-Parallel Finite-State Machines* (2014), and was
ported from the Haskell-Works `hw-json-simd` project. See [parsing/json.md](../parsing/json.md)
and [parsing/json-index.md](../parsing/json-index.md).

## Consequences

**Positive:**

- The portable path avoids the branch-misprediction cost of a scalar FSM: the PFSM's lookup
  tables fit in L1 and are branch-prediction-friendly, measuring **+22–43% over the scalar
  reference** on x86_64 ([optimizations/history.md](../optimizations/history.md)).
- The table-driven PFSM needs no special CPU features, so the fallback is fast on any target.
- On the platforms most users run, SIMD classification pushes throughput higher still: the AVX2
  standard-cursor path measures ~730 MiB/s on Zen 4, and a PFSM + BMI2 configuration reaches
  ~880 MiB/s — both roughly **40–77% faster than scalar** parsing
  ([parsing/json.md](../parsing/json.md)).

**Negative / trade-offs:**

- The packed-`u32` transition tables are opaque; modifying the parser means understanding the
  table-packing scheme rather than reading straightforward control flow.
- Several parsing paths (simple/standard cursors, per-arch SIMD, PFSM) must be kept behaviorally
  identical and tested in parallel.

**Alternatives considered and rejected:**

- A **batched PFSM** variant was ~25% slower and not deployed.
- A **NEON PFSM shuffle** approach on ARM was reverted (−47%): sequential phi extraction negated
  the parallel gains. Both are recorded in [optimizations/history.md](../optimizations/history.md).
