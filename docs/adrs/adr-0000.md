# ADR-0000: Use Architecture Decision Records

## Status

✅ Accepted

## Context

Succinctly has accumulated a substantial body of design decisions — the choice of
semi-indexing over DOM parsing, the PFSM JSON parser, the YAML oracle, and a long series of
optimizations that were tried and either accepted or rejected. Until now these decisions have
been recorded **informally**: in the P1–P12 / O1–O3 "accepted/rejected optimization" log
inside [CLAUDE.md](../../CLAUDE.md), in the narrative optimization journey in
[parsing/yaml.md](../parsing/yaml.md), and in [optimizations/](../optimizations/).

That record is valuable, but it is not a versioned, status-tracked decision corpus. There are
no stable identifiers to cite, no status lifecycle (proposed → accepted → superseded), no
explicit supersession links, and the material is interleaved with agent instructions rather
than standing on its own. As a result, decisions get revisited without their original
trade-offs, and newcomers lack a single place to learn *why* the system is shaped the way it is.

We need a lightweight, version-controlled method for recording significant decisions —
including the decisions to *not* pursue an approach, which are some of the most valuable
history this project has.

## Decision

We will use Architecture Decision Records (ADRs) to document significant architectural and
design decisions for succinctly.

Each ADR follows the format popularized by Michael Nygard:

- **Title** — a short noun phrase describing the decision (e.g., "ADR-0001: Semi-Indexing over
  DOM Parsing"). A decision to reject an approach is phrased as "Reject X".
- **Status** — one of `Proposed`, `Accepted`, `Deprecated`, or `Superseded`, shown with the
  emoji from the [inventory legend](README.md#status-legend). A decision to reject an approach
  is `Accepted` — the decision *to reject* is what is in effect.
- **Context** — the forces at play (technical, organizational, project-specific).
- **Decision** — the response, stated in active voice ("We will ..." / "We will not ...").
- **Consequences** — the resulting context: positive, negative, and neutral outcomes.

ADRs will be:

- Stored in `docs/adrs/` within the repository, versioned alongside the code they describe.
- Written in Markdown, with the `# ADR-NNNN: Title` heading as the **first line** of the file
  and no YAML front-matter, so the `update-adr-inventory` tooling can parse them reliably.
- Numbered sequentially with zero-padded four-digit identifiers (`adr-0001.md`, `adr-0002.md`).
  Numbers are never reused; superseded ADRs remain in the repository for historical reference.
- Kept concise — ideally one to two pages.
- Scoped to a single decision per record.

The inventory table in [README.md](README.md) is regenerated by the
[`update-adr-inventory`](../../.claude/skills/update-adr-inventory/SKILL.md) skill, which reads
each ADR's title and status and derives the date from git. The
[`review-adr`](../../.claude/skills/review-adr/SKILL.md) skill checks new and existing ADRs
against this format.

## Consequences

- Every significant decision — including rejected optimizations — has a stable, citable ID and
  a written record, so contributors can understand the rationale behind the current design.
- ADRs complement the [knowledge map](../index.md): the wiki explains *how it works today*;
  ADRs record *why we chose it and what we rejected*.
- Superseded decisions stay in the repository, preserving how the architecture evolved.
- There is a small overhead in writing and maintaining ADRs, offset by less repeated
  discussion and faster onboarding.
- The existing P-series optimization log in [CLAUDE.md](../../CLAUDE.md) remains as day-to-day
  agent context, but the ADRs become the canonical, linkable record of the underlying decisions.
