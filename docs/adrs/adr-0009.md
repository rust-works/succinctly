# ADR-0009: Reject a Parse-Time Newline Index (P7)

## Status

✅ Accepted

## Context

JSON has a `NewlineIndex` structure that maps byte offsets to line/column positions.
Optimization **P7** asked whether YAML should build an equivalent newline index during parsing,
on the assumption that it was a parsing optimization worth porting.

Investigating first revealed that the premise was wrong. JSON's `NewlineIndex` is **not** built
during parsing at all — it is constructed on demand only by the `jq-locate` CLI tool to convert
`--line X --column Y` into a byte offset. It has zero impact on JSON parsing, queries, or
benchmarks. YAML, meanwhile, already computes line numbers lazily via an O(n) `current_line()`
that is called only on error paths (4 call sites, all in error reporting), so valid documents
never pay for it.

## Decision

We will **not** build a newline index during YAML parsing.

Doing so would add an O(n) cost to the **hot path** (parsing every document) purely to speed up a
**cold path** (error reporting and an optional CLI feature) that valid-document workloads never
exercise. That is the wrong trade: the benchmarks parse valid YAML, so a parse-time index would
be pure overhead with no benefit. The correct framing is that a newline index is a **CLI/UX
feature, not a parsing optimization** — so if line/column lookup is wanted, it should be built
**on demand** by the tool that needs it, not during parsing. Rejected 2026-01-17.

Full analysis: [parsing/yaml.md § P7](../parsing/yaml.md#p7-newline-index---rejected-).

## Consequences

- YAML parsing stays free of a newline index; line numbers remain lazily computed on error paths
  only.
- **The on-demand CLI path was subsequently taken, and does not contradict this ADR.** A
  `yq-locate` tool was added (optimization P11), and P12-A later made even the JSON-style newline
  index lazy via `OnceCell`, so the index is built only when a locate query needs it — exactly the
  "on demand, not during parsing" resolution this ADR endorses. This ADR rejects *parse-time*
  indexing specifically; it does not reject line/column lookup as a feature.
- Records the reusable distinction: **not every JSON feature is a parsing optimization** — some
  (like `NewlineIndex`) are CLI ergonomics that belong on the cold path.
