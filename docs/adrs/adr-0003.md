# ADR-0003: Oracle Parser for YAML

## Status

✅ Accepted

## Context

Semi-indexing (ADR-0001) depends on being able to identify a document's structure in a single
scan. For JSON this is easy: `{`, `}`, `[`, `]`, `:`, `,`, and `"` are unambiguous structural
characters, so a data-parallel scanner (ADR-0002) can classify bytes with almost no context.

YAML is fundamentally harder. Its structure is **indentation-based** rather than
bracket-delimited, and almost every significant character is context-sensitive:

- `:` may separate a key from a value, or be a literal colon inside a plain scalar or URL.
- `-` may start a block-sequence item, or be a minus sign, or part of a plain scalar.
- `#` starts a comment only when preceded by whitespace.
- `|` and `>` introduce block scalars whose bodies are governed by indentation and chomping.
- `&`/`*` are anchors/aliases; `"` and `'` are two different quoting styles with different
  escaping rules; flow style (`{}`, `[]`) nests JSON-like syntax inside block context.

A stateless SIMD classifier cannot resolve these — the meaning of a character depends on
indentation, flow depth, and quoting context accumulated from the start of the line/document.
See the character-ambiguity analysis in [parsing/yaml.md](../parsing/yaml.md).

## Decision

We will parse YAML with an **oracle + index** model: a sequential "oracle" resolves YAML's
context-sensitive grammar and emits the semi-index, after which all queries run oracle-free.

- **Oracle (sequential):** a single-pass parser that tracks indentation and flow context,
  disambiguates each character, and emits the interest bits (IB), balanced-parentheses (BP), and
  **type bits (TY)** that make up the index. TY is needed for YAML but not JSON because, in YAML,
  the byte at a structural position does not by itself reveal the node's type.
- **Semi-index (O(1) queries):** once built, navigation uses only the BP tree structure and the
  bit vectors — the oracle never runs again.

The oracle handles block style, flow style, anchors, aliases, and block scalars uniformly. It is
implemented in [src/yaml/parser.rs](../../src/yaml/parser.rs) with the architecture documented in
[src/yaml/mod.rs](../../src/yaml/mod.rs) and [parsing/yaml-index.md](../parsing/yaml-index.md).

## Consequences

**Positive:**

- Correctly indexes real-world YAML (Kubernetes manifests, CI configs, multi-document streams)
  including the ambiguous constructs a stateless scanner cannot handle.
- Preserves the semi-indexing payoff: the expensive context resolution happens once,
  sequentially; every subsequent query is O(1) over the index. The YamlIndex index-build
  throughput is ~250–400 MiB/s, and `succinctly yq` runs 1.7–10.6× faster than `yq` on Apple
  Silicon and ~15–23× faster on x86_64 (AMD Ryzen 9 7950X), with a fraction of its memory (see
  [benchmarks/yq.md](../benchmarks/yq.md)).

**Negative / trade-offs:**

- The oracle is inherently **sequential** — it cannot be data-parallelized the way JSON
  structural scanning can, because each character's meaning depends on prior context. This makes
  YAML's index-build slower per byte than JSON's, and it constrains which SIMD optimizations can
  apply (see ADR-0006, ADR-0008).
- The oracle is the most complex component in the codebase: it must encode YAML's grammar
  faithfully, and correctness bugs there surface as wrong query results rather than parse errors.

**Neutral:**

- The oracle emits into the same BitVec + BalancedParens substrate as JSON and DSV, so
  navigation, `find_close`, and the jq evaluator are shared across formats.
