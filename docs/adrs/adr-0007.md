# ADR-0007: Reject the Flow-Collection SIMD Fast Path (P5)

## Status

✅ Accepted

## Context

YAML supports JSON-like *flow collections* — inline `{...}` maps and `[...]` sequences.
Optimization **P5** proposed a SIMD fast path to scan these collections in bulk. Micro-benchmarks
were promising: SIMD showed 8–14× wins for flow collections of 64–128 bytes.

By this point three optimizations (P2.6, P2.8, P3 — ADR-0004/0005/0006) had already been rejected
because micro-benchmark wins failed to survive end-to-end measurement. So before implementing P5,
we checked whether the micro-benchmark's input sizes matched reality.

## Decision

We will **not** implement the flow-collection SIMD fast path. This decision was made at the
**analysis stage, before writing any code**, by applying the lesson from P2.6/P2.8/P3.

The SIMD approach only starts winning at ~32+ bytes, but real YAML flow collections are typically
**10–30 bytes** — roughly 90% of them fall below the size where SIMD pays off. The 8–14× wins in
the micro-benchmark came from 64–128-byte inputs that essentially do not occur in practice, so the
optimization would have added complexity and a likely regression on real workloads for no real
gain. Rejected 2026-01-17.

Full analysis: [parsing/yaml.md § P5](../parsing/yaml.md#p5-flow-collection-fast-path---rejected-).

## Consequences

- Flow collections continue to use the existing scalar/early-exit scanning path.
- Demonstrates the payoff of the earlier rejections: the guideline "**micro-benchmark wins only
  translate when the benchmarked input sizes exist in real workloads**" let us avoid the
  implement-benchmark-revert cycle entirely and reject at analysis.
- Neutral: this ADR records a *non*-implementation. If flow-collection workloads with large inline
  collections ever become common, the decision can be revisited with real-world size data.
