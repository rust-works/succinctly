# ADR-0006: Reject Branchless Character Classification (P3)

## Status

✅ Accepted

## Context

The YAML oracle (ADR-0003) classifies characters constantly — is this byte a structural marker,
whitespace, a quote, an indicator? These checks are written with `matches!` macros, which compile
to conditional branches. Optimization **P3** proposed replacing them with 256-byte lookup tables
indexed by the byte value, to eliminate the branches. Micro-benchmarks of classification in
isolation showed a 3–29% improvement.

## Decision

We will **not** replace `matches!`-based classification with lookup tables.

End-to-end benchmarks showed a **catastrophic regression** on the most common workload — simple
key/value parsing:

| Workload       | Change |
|----------------|--------|
| simple_kv/10   | +25%   |
| simple_kv/100  | +44%   |
| simple_kv/1000 | +43%   |
| simple_kv/10000| +43%   |

Two effects dominated any branch-elimination benefit: modern branch predictors already classify
these characters with 93–95% accuracy (the branches are highly predictable), and the 256-byte
tables polluted the cache while `.map_or()` added its own overhead. The net result was 25–44%
slower parsing. Rejected 2026-01-17.

Full analysis: [parsing/yaml.md § P3](../parsing/yaml.md#p3-branchless-character-classification---rejected-).

## Consequences

- Character classification stays branch-based (`matches!`).
- Completes the P2.6 / P2.8 / P3 trio: three consecutive optimizations where micro-benchmarks
  misled. The codified guideline — **prefer predictable branches over lookup tables for hot
  character classification; validate against end-to-end benchmarks** — is what enabled rejecting
  P5 (ADR-0007) before implementing it.
- Negative: none — the change was reverted; the value is the recorded evidence.
