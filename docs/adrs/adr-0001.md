# ADR-0001: Semi-Indexing over DOM Parsing

## Status

✅ Accepted

## Context

Succinctly's purpose is to run queries (jq/yq-style) over JSON, YAML, and DSV documents. The
conventional approach — used by serde_json, simd-json, jq, and yq — is to parse the whole
document into an in-memory tree (a DOM), then walk that tree. A DOM materializes every value
whether or not the query touches it, so memory scales with the size of the *document* rather
than the size of the *result*.

The observation that motivates a different design: **most queries access only a small fraction
of a document.** A jq path like `.users[3].name` touches a handful of bytes in a multi-megabyte
file, yet a DOM parser allocates the entire structure first. For large documents and selective
queries this is almost pure waste — in both allocation and the parsing work to produce values
that are never read.

## Decision

We will use **semi-indexing** rather than DOM parsing as the core architecture for all three
formats.

Semi-indexing builds a lightweight structural index of a document in a single pass, then
navigates and extracts values lazily:

1. **Scan** — a (SIMD-accelerated) pass identifies structural characters (braces, brackets,
   quotes, newlines).
2. **Encode** — structural boundaries become a [balanced-parentheses](../architecture/balanced-parens.md)
   bitvector plus auxiliary "interest bit" vectors, giving a compact map of the document's
   nesting.
3. **Navigate** — tree operations (parent, child, sibling, skip-subtree) run in O(1) via
   rank/select on the BP bitvector, with no re-parsing.
4. **Extract** — a value is parsed only when a query actually reads it.

The index adds ~3–6% memory overhead on top of the source bytes, versus the 100%+ overhead of
building a full DOM. See [architecture/semi-indexing.md](../architecture/semi-indexing.md) for
the full design and [parsing/json-index.md](../parsing/json-index.md) /
[parsing/yaml-index.md](../parsing/yaml-index.md) for the per-format encodings.

This decision rests on prior art: Ottaviano et al., *Semi-Indexing Semi-Structured Data in Tiny
Space* (2011), and the Haskell-Works heritage documented in
[architecture/prior-art.md](../architecture/prior-art.md).

## Consequences

**Positive:**

- Dramatically lower memory: roughly **18–46× less** than serde_json / simd-json, and **5–25×
  less** than jq, because unread values are never materialized. See [benchmarks/](../benchmarks/).
- Faster selective queries: index-build is a single linear pass, and navigation is O(1), so
  work is proportional to what the query touches, not the document size.
- Streaming output without building an intermediate tree.

**Negative / trade-offs:**

- The index is an upfront cost paid even when a query is trivial; for very small documents
  (< 1 KB) or when the *entire* document is consumed, a plain DOM parser can be simpler and
  competitive.
- Semi-indexing performs **minimal validation** compared to full parsers. Benchmark comparisons
  to jq/yq therefore reflect both the architectural advantage *and* reduced validation work —
  this caveat is called out in [CLAUDE.md](../../CLAUDE.md) and the root README so the numbers
  are not read as a pure like-for-like parser comparison.
- The implementation (rank/select, balanced parentheses, per-format oracles) is more complex
  than a straightforward recursive-descent DOM parser.

**Neutral:**

- All three format modules (`json/`, `yaml/`, `dsv/`) share the same BitVec + BalancedParens
  foundation, so the investment in succinct data structures is amortized across formats.
